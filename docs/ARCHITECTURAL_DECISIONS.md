# ADR-001  
## Multi-Tenant Architecture

Status: Accepted

Context:

Tatame Pro must support multiple organizations operating independently inside the platform.

Problem:

Without strict isolation, data belonging to one organization could leak into another tenant context.

Decision:

The platform uses strict multi-tenant architecture with tenant-scoped entities.

All tenant data must contain:

tenant_id

Backend services must enforce tenant filtering.

Consequences:

All queries must validate tenant scope.

Cross-tenant joins must be avoided.

All prompts generated by Claude must preserve tenant isolation.

---

# ADR-002  
## Backend Authority Over Business Logic

Status: Accepted

Context:

Frontend applications can be manipulated and cannot be trusted as a source of truth.

Problem:

If business logic exists only on the frontend, malicious clients could bypass validation.

Decision:

All critical operations must be validated at backend level.

Frontend validation is only a user experience enhancement.

Consequences:

Backend services must enforce:

role validation  
tenant isolation  
state transitions  

Frontend must not implement critical business rules.

---

# ADR-003  
## Additive Database Migration Strategy

Status: Accepted

Context:

Tatame Pro must preserve historical records for compliance and traceability.

Problem:

Schema changes can destroy data or break production flows if destructive migrations are executed.

Decision:

Database migrations must follow additive patterns.

Allowed operations:

add tables  
add columns  
add indexes  

Forbidden operations:

drop tables  
truncate historical data  
remove critical columns without migration plan  

Consequences:

All schema evolution must preserve historical integrity.

Claude must reject prompts proposing destructive migrations.

---

# ADR-004  
## Immutable Historical Records

Status: Accepted

Context:

Certain data must represent historical truth.

Examples:

graduations  
diplomas  
competition results  

Problem:

If historical records are modified or overwritten, institutional credibility is compromised.

Decision:

These records must become immutable after confirmation.

Corrections must occur through new records or audit entries.

Consequences:

System must preserve auditability.

Historical records must not be silently edited.

---

# ADR-005  
## AI-Assisted Development Workflow

Status: Accepted

Context:

Tatame Pro development is assisted by AI agents.

Actors include:

Claude (analysis and prompt generation)  
Lovable (code generation and execution)

Problem:

Unstructured AI-generated code can introduce architectural instability.

Decision:

AI development must follow a governed workflow.

Claude analyzes repository first.

Claude generates deterministic prompts.

Lovable executes prompts.

Human architect validates results.

Consequences:

All development must follow the AI engineering workflow.

Prompts must be deterministic.

Claude must not generate free-form code instructions.

---

# ADR-006  
## Domain-Driven Product Architecture

Status: Accepted

Context:

Tatame Pro operates in a complex sports governance domain.

Problem:

Without domain boundaries, features can become entangled and difficult to maintain.

Decision:

System architecture follows domain separation.

Key domains include:

identity  
athletes  
memberships  
graduations  
events  
rankings  
billing  
governance  

Consequences:

Modules must respect domain boundaries.

Cross-domain mutation must be minimized.

Service layer must coordinate interactions.

---

# ADR-007  
## Public Verification Model

Status: Accepted

Context:

Tatame Pro must allow public verification of official credentials.

Examples include:

digital athlete cards  
diplomas  
certificates  

Problem:

Public verification must not expose private user data.

Decision:

Public verification uses token-based endpoints.

Public tokens reference documents but expose only non-sensitive fields.

Consequences:

Verification endpoints must sanitize data before returning results.

Private data must remain protected.

---

# ADR-008  
## Observability and Auditability

Status: Accepted

Context:

The platform must remain diagnosable and secure.

Problem:

Without observability, diagnosing operational problems becomes difficult.

Decision:

Tatame Pro includes observability and audit layers.

These include:

audit logs  
security events  
system health monitoring  

Consequences:

Critical actions must be logged.

Security-sensitive operations must generate audit events.

---

# ADR-009  
## Superadmin Impersonation Model

Status: Accepted

Context:

Support staff must occasionally access tenant environments for troubleshooting.

Problem:

Direct access without traceability creates security risks.

Decision:

Impersonation must be explicit and auditable.

All impersonation sessions must be recorded.

Consequences:

Impersonation requires explicit start and end.

Audit records must track impersonation events.

---

# ADR-010  
## Canonical Product Center

Status: Accepted

Context:

Tatame Pro must maintain clear product identity.

Problem:

Feature creep can push the product toward unrelated domains.

Decision:

The product center is defined as:

sports governance  
athlete affiliation  
certification and progression  
document verification  

Consequences:

New features must strengthen the canonical center.

Features unrelated to this center must be rejected or classified as adjacent modules.
