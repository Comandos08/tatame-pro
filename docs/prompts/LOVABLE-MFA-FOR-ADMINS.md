# LOVABLE PROMPT — MFA/TOTP for Admin Roles

> **Type:** Security feature — new capability
> **Priority:** P1 (required before sales/PRD)
> **Scope:** Frontend (auth flow) + Edge Function (enforcement) + migration (preference flag)

---

## 1. OBJECTIVE

Require Time-based One-Time Password (TOTP) MFA for any user with role
`SUPERADMIN_GLOBAL` or `ADMIN_TENANT`. Use Supabase Auth's native MFA so
the secret is stored server-side and the QR code is generated by Supabase.

---

## 2. SYSTEM CONTEXT

- Auth provider: Supabase Auth.
- Roles: `SUPERADMIN_GLOBAL`, `ADMIN_TENANT`, `FEDERATION_ADMIN`, `COACH`,
  `ATLETA`, `STAFF` — defined in `user_roles` table.
- Auth state machine: `src/contexts/AuthContext.tsx` + identity wizard.
- Existing session timeout: `src/hooks/useSessionTimeout.ts` (30 min).

Supabase MFA primitives to use:
- `supabase.auth.mfa.enroll({ factorType: 'totp' })`
- `supabase.auth.mfa.challenge({ factorId })`
- `supabase.auth.mfa.verify({ factorId, challengeId, code })`
- `supabase.auth.mfa.unenroll({ factorId })`

---

## 3. CURRENT STATE

No MFA implementation in the codebase (`grep MFA|2FA|totp` returns no
matches in `src/`). Admin users authenticate with email + password only.

---

## 4. EXPECTED BEHAVIOR

### 4.1 Enrollment (first login as admin)

After successful password login, if the user's `aal` (authenticator
assurance level) is `aal1` AND they have any admin role:

1. Redirect to `/security/mfa/enroll`
2. Show QR code from `enroll().data.totp.qr_code`
3. User scans with authenticator app, enters 6-digit code.
4. On verify success, set `aal2` and proceed to original target.
5. Persist `mfa_enrolled_at` on `profiles` table for audit.

The user CANNOT skip enrollment if they have an admin role. Lower roles
(ATLETA, STAFF) see an opt-in banner but are not forced.

### 4.2 Login (subsequent sessions)

After password login, if `aal === 'aal1'` and the user already enrolled:

1. Redirect to `/security/mfa/challenge`
2. Show 6-digit code input.
3. On verify success, proceed.
4. On 5 wrong attempts in 15 min, reuse the existing `account-lockout.ts`
   pattern.

### 4.3 Recovery codes

On enrollment, generate 8 single-use recovery codes via a new edge function
`generate-mfa-recovery-codes`. Store SHA-256 hashes in a new table
`mfa_recovery_codes`. Show codes ONCE; user must download/print.

### 4.4 Admin disable (escape hatch)

A new edge function `superadmin-reset-mfa` (callable by SUPERADMIN_GLOBAL
only) allows disabling MFA for a user who lost their device + recovery
codes. Logs to `audit_logs` with type `MFA_RESET`.

## 6. CONSTRAINTS

- Reuse existing rate-limiter (`secure-rate-limiter.ts`) — preset
  `mfaChallenge` = 10 attempts / 15 min / IP.
- Reuse existing `account-lockout.ts` pattern for repeated failures.
- Reuse existing audit logger (`audit-logger.ts`) with new event types:
  `MFA_ENROLLED`, `MFA_VERIFIED`, `MFA_FAILED`, `MFA_RESET`,
  `MFA_RECOVERY_CODE_USED`.
- Recovery codes table MUST have RLS: only the owning user can SELECT
  their own hashed codes. SUPERADMIN can DELETE for the reset flow.
- All new edge functions MUST use `corsPreflightResponse(req)` and the
  A07 error envelope.

---

## 7. ACCEPTANCE CRITERIA

1. Login as ADMIN_TENANT user without MFA → forced to enroll on first
   post-login navigation.
2. Logout + login again → challenge page appears.
3. 5 wrong codes in 15 min → account locked (existing lockout flow).
4. Recovery code accepted; consumes the code and bumps `aal` to `aal2`.
5. SUPERADMIN_GLOBAL can disable MFA for a target user via
   `superadmin-reset-mfa` (audit row `MFA_RESET` is written).
6. Existing E2E suite still passes.
7. New E2E spec `e2e/security/mfa-flow.spec.ts` covers enroll +
   challenge + recovery + lockout.

---

## 8. RISK ANALYSIS

| Risk | Mitigation |
|---|---|
| Locking out the only superadmin during rollout | Ship the `superadmin-reset-mfa` first AND ensure at least 2 superadmin users exist before forcing the gate. |
| Confusing users with sudden MFA prompt | Add a banner 7 days ahead of enforcement; document in `docs/runbooks/`. |
| Recovery codes leaking | Hash with SHA-256, never store plaintext, single-use. |

---

## 9. ROLLBACK

If forced enrollment misbehaves: feature-flag the guard via a new
`tenant_flags` row `mfa_enforcement_enabled` (default false during
rollout). Toggle off to bypass the redirect; users stay enrolled.
